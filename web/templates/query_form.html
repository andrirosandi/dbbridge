Variable Dictionary / Cheat Sheet

You can use the following dynamic variables in your SQL:

  • {param} - Standard parameter. User will be prompted to enter a value.
  • {param:default_value} - Parameter with default value. Used when no value is provided.
    Example: {itemid:%} defaults to % (matches all).
  • {pagination} - Adds pagination logic automatically. Controls page and per_page.
  • {pagination:1:20} - Pagination with custom default (Page 1, Limit 20).
  • {pagination::100} - Pagination with default Limit 100.
  • {order_by:column_name} - Dynamic ORDER BY with whitelist validation.
    Example: {order_by:created_at(id,name):desc} - default column is created_at, allowed columns are id and name, default direction is desc.
  • {select}col1,col2{endselect} - Metadata block. Use this to enable total row count.
    Columns inside this block will be used for the main query. The COUNT(*) query replaces this block.
  • {ids} - Arrays/Lists are supported! IN ({ids}) expands automatically to IN (?, ?, ?).
    • UI Input: Type as JSON string: [1, 2, 3] or ["a", "b"]
    • API JSON: Send as actual array: "ids": [1, 2, 3]

Example Query:

SELECT {pagination::50} {select}itemid, description{endselect}
FROM initem
WHERE itemid LIKE {itemid:%}
{order_by:itemid(itemid,description):asc}

This query supports:

  • Pagination (default 50 items per page)
  • Total count (via {select}...{endselect} block)
  • Search by itemid (default: % = all)
  • Sorting by itemid or description (default: itemid ASC)
Note: JSON body parameters page and per_page override defaults. Use order_by and order_direction params to control sorting.

Reserved Parameter Names:

The following names CANNOT be used as user-defined query parameters:

page, per_page, order_by, order_direction
